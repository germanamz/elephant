# Tusk Integration — Phase 2: Task Creation, Player Registration & Service Access

## Prerequisites

- **Phase 1 completed**: `internal/work/` package exists with `Engine`, `Config`, `NewEngine()`, `Close()`, WBS level constants, and `ValidLevel()`. The Tusk SDK dependency is in `go.mod`.

## Inherits From

Phase 1 introduced:
- `internal/work/engine.go` — `Engine` struct wrapping `*tusk.Client`, `Config` with `DBPath`, `NewEngine(cfg)` constructor, `Close()` method.
- `internal/work/level.go` — `LevelVision`, `LevelRoadmap`, `LevelInitiative`, `LevelStory`, `LevelTask`, `LevelSubtask` constants and `ValidLevel(string) bool`.
- Hardcoded Tusk config: workflow `"wbs"` (`pending → active → completed → deleted`), project `"default"`.

The implementer can rely on `NewEngine` creating a fully initialized Tusk client with the correct workflow and project.

## Goal

Add task creation helpers (with WBS level enforcement), player registration, and service accessor methods to the `Engine`. At the end of this phase, the `internal/work/` package is feature-complete per the design spec — callers can create WBS-shaped tasks, register agents as players, and access Tusk services for operations like claim, complete, pop, and annotate.

## Reference Material

- Design spec: `docs/superpowers/specs/2026-04-09-tusk-integration-design.md`
- Phase 1 doc: `docs/superpowers/specs/tusk-integration-phase-1.md`
- Tusk SDK task creation: `tusk.Client.Tasks.Create(ctx, *domain.Task) error` — the task must have `ProjectID` set; ID and ShortID are auto-generated by Tusk.
- Tusk SDK player registration: `tusk.Client.Players.Register(ctx, playerID, playerType) (*domain.Player, error)` — `playerType` must be `"human"` or `"agent"`.
- Tusk service types: `github.com/germanamz/tusk/service` — `TaskService`, `PlayerService`, `RelationService`.
- Tusk domain UDA: `domain.Task.UDA` is `map[string]any` — values must be strings per Tusk validation.

## Tasks

### Task 1: Add `CreateTask` to Engine

Add to `internal/work/engine.go` (or a new file `internal/work/task.go` if `engine.go` would exceed ~100 lines):

```go
// CreateTaskParams holds the inputs for creating a WBS-shaped task.
type CreateTaskParams struct {
    Title       string
    Description string
    Level       string     // must pass ValidLevel()
    ParentID    *uuid.UUID // optional parent for hierarchy
}

// CreateTask creates a task in Tusk with the correct WBS level UDA and project assignment.
func (e *Engine) CreateTask(ctx context.Context, params CreateTaskParams) (*domain.Task, error)
```

**Behavior:**
1. Validate `params.Level` with `ValidLevel()`. If invalid, return `fmt.Errorf("work: create task: invalid WBS level %q", params.Level)`.
2. Construct a `domain.Task` with:
   - `Title`: `params.Title`
   - `Description`: `params.Description`
   - `ProjectID`: `"default"` (use the `DefaultProject` constant from Phase 1)
   - `ParentID`: `params.ParentID`
   - `UDA`: `map[string]any{"wbs_level": params.Level}`
3. Call `e.client.Tasks.Create(ctx, &task)`.
4. On error, wrap: `fmt.Errorf("work: create task: %w", err)`.
5. Return the task (Tusk populates `ID`, `ShortID`, `CreatedAt`, etc. in place).

**Acceptance:** A task created with `Level: "story"` has `UDA["wbs_level"] == "story"` and `ProjectID == "default"`. A task with `ParentID` set has the correct parent linkage. An invalid level returns an error without calling Tusk.

### Task 2: Add `RegisterAgent` to Engine

Add to `internal/work/engine.go` (or `internal/work/player.go`):

```go
// RegisterAgent registers an agent as a Tusk player.
func (e *Engine) RegisterAgent(ctx context.Context, agentID string) (*domain.Player, error)
```

**Behavior:**
1. Call `e.client.Players.Register(ctx, agentID, "agent")`.
2. On error, wrap: `fmt.Errorf("work: register agent: %w", err)`.
3. Return the player.

**Acceptance:** Registered player has `Type == "agent"` and the given `ID`. Calling `RegisterAgent` twice with the same ID behaves according to Tusk's semantics (verify what happens — likely returns the existing player or an error; test and document the behavior).

### Task 3: Add service accessor methods to Engine

Add to `internal/work/engine.go`:

```go
// Tasks returns the underlying Tusk TaskService for operations
// that don't need Elephant-specific wrapping (claim, complete, pop, annotate, etc.).
func (e *Engine) Tasks() *service.TaskService

// Players returns the underlying Tusk PlayerService.
func (e *Engine) Players() *service.PlayerService

// Relations returns the underlying Tusk RelationService.
func (e *Engine) Relations() *service.RelationService
```

Each method returns the corresponding field from `e.client` (e.g., `e.client.Tasks`, `e.client.Players`, `e.client.Relations`).

**Acceptance:** Callers can use `engine.Tasks().Claim(...)`, `engine.Tasks().Pop(...)`, etc. directly. The accessor methods compile and return non-nil services after `NewEngine()`.

### Task 4: Unit tests for `CreateTask`

Create or extend `internal/work/engine_test.go` (or `internal/work/task_test.go`):

- `TestCreateTask_ValidLevel`: create a task with each of the six WBS levels. Verify `UDA["wbs_level"]` is set correctly, `ProjectID` is `"default"`, and `ID`/`ShortID` are populated.
- `TestCreateTask_InvalidLevel`: pass an invalid level string. Verify error is returned and contains the invalid level. Verify no task was created (list tasks, expect empty or unchanged).
- `TestCreateTask_WithParent`: create a parent task, then a child task with `ParentID` set. Verify the child's `ParentID` matches the parent's `ID`.
- `TestCreateTask_EmptyTitle`: verify Tusk's behavior when title is empty — document whether it errors or allows it; test accordingly.

**Acceptance:** All tests pass using real Tusk client + temp SQLite DB.

### Task 5: Unit tests for `RegisterAgent` and service accessors

Create or extend test files:

- `TestRegisterAgent_Success`: register an agent, verify `Type == "agent"` and `ID` matches.
- `TestRegisterAgent_Duplicate`: register the same agent ID twice. Document and test the actual behavior (error or idempotent return).
- `TestServiceAccessors`: after `NewEngine()`, verify `Tasks()`, `Players()`, and `Relations()` return non-nil. Perform a basic operation through each (e.g., `Tasks().List(ctx, nil)` returns empty slice, `Players().List(ctx)` returns empty or single-element slice).

**Acceptance:** All tests pass. Service accessors are usable for downstream operations.
